# ADR-002: 正規表現エンジンに RE2 を採用する

- **ステータス:** Accepted
- **決定日:** 2026-07-14

## コンテキスト

要件定義書 §6 で正規表現検索が必須。数 GB のログファイル検索を要求 (§5, §8) するため、**線形時間で動作し ReDoS 耐性のあるエンジン**が必要。

`std::regex` は MSVC 実装が遅く、また NFA バックトラック方式のため悪意ある入力で指数時間になる可能性があり不採用。

## 選択肢

1. **Google RE2**
2. Intel Hyperscan
3. PCRE2 (JIT)
4. 自作 (Thompson NFA)

## 決定

**RE2 を単独採用する。**

## 根拠

| 項目 | RE2 | Hyperscan | PCRE2-JIT |
|---|---|---|---|
| 線形時間保証 | ✅ | ✅ | ❌ (バックトラック) |
| ライセンス | BSD-3 | BSD-3 | BSD-like |
| バイナリサイズ | 小 (~1-2MB 静的) | 大 (~10MB 静的) | 中 |
| 依存 | Abseil (`absl`) | AVX2 前提、Boost 一部 | JIT ライブラリ |
| API 使いやすさ | ○ | △ (規則コンパイル API 独特) | ○ |
| 後方参照 | ✗ | ✗ | ✓ |
| 一般用途性能 | 十分 | 極高 | 高 |

- **要件との整合:** 一般的なテキスト/ログ検索は RE2 で十分 (数 100MB/s 出る)
- **バイナリ肥大回避:** 起動 0.3s / メモリ 20MB を圧迫しない
- **AVX2 非依存:** 古い Windows 10 マシンでも動作
- **Abseil 依存:** すでに他モジュールでも活用検討 (`absl::flat_hash_map` 等)

## 影響

- **後方参照 (`\1` 等) は非対応** → UI 側で警告し、必要な場合に別実装フォールバック検討
- Abseil をサブモジュール取り込み or FetchContent で導入
- 検索クエリは RE2 の `RE2::Options` にマップする薄いラッパを設ける

## 却下理由

- **Hyperscan:** 性能は最高だが、単一パターン検索が主用途の当ソフトではオーバースペック。AVX2 前提と 10MB のバイナリ膨張が要件と噛み合わない
- **PCRE2-JIT:** 高速だが ReDoS 耐性がなく、複雑パターンで指数時間リスク。要件書の「巨大ファイル快適動作」に反する可能性
- **自作:** 保守コストが割に合わない

## 将来の再評価

- Grep で 10 万件級パターンを扱う需要が出た場合、Hyperscan の追加採用を再検討する (詳細設計 §7 「両方搭載」案)
